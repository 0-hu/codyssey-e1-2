## 2. 코드 및 클래스 설계 원칙

본 프로그램은 클래스 기반으로 책임을 분리하여 작성했습니다. 
전역 변수 대신 **클래스의 멤버 변수**를 활용해 상태(State)를 추적하여, 의존성을 낮추고 유지 보수의 편의를 높였습니다.

### 🧩 Quiz와 QuizGame 클래스 책임 분리 (SRP 관점)
- **`Quiz` 클래스 (`quiz.py`)**: 단일 "문제 단위"의 모델 데이터 구조체입니다. 문제 지문, 4개의 선택지 리스트, 정답 번호, 힌트 정보를 갖습니다. 사용자 입력을 받아 자신이 갖고 있는 정답과 비교하는 `check_answer`를 담당합니다.
- **`QuizGame` 클래스 (`quiz_game.py`)**: 전체 프로그램의 컨트롤러입니다. 프로그램의 메인 루프(`run`), I/O 메시지(`show_menu`), 데이터 관리(`load_data`, `save_data`), 그리고 퀴즈 모델 인스턴스들의 컬렉션을 리스트로 들고 있으며 게임 흐름 로직을 통합 관리합니다.

### ⚙️ 로직 분리 기준
`QuizGame` 내부에서도 역할을 메서드 단위로 쪼개어 가독성을 높였습니다.
1. **공통 입력 유효성 처리 로직**: `get_valid_input()`
2. **콘솔 I/O와 게임 루틴 로직**: `show_menu()`, `play_quiz()`, `add_quiz()`, `list_quizzes()`, `delete_quiz()`, `show_score()`
3. **영구 저장 데이터 매니지먼트**: `load_data()`, `save_data()`, `get_default_quizzes()`

### 🛠️ 요구사항 변경 시 수정 범위 국소화 
역할을 나누어 변경 사항 발생 시 코드 영향 범위를 최소화합니다.
- 점수 규칙이 100점 만점이 아니라 개수 제도로 바뀌었다면? → `QuizGame.play_quiz()` 안의 로직만 수정.
- 오지선다(선택지 5개)로 바뀐다면? → `Quiz` 클래스와 UI 출력부만 수정.

## 3. 안정성과 예외/오류 처리 기법

사용자의 예상 밖 행동에도 프로그램이 충돌(Crash) 나지 않도록 구성하였습니다.

### 🛡️ 입력 예외/오류 케이스 방어
`get_valid_input(prompt, min, max)` 공통 메서드 내에서 모든 방어 로직을 통합 처리합니다.
- **공백 입력 오류 방지**: `.strip()`을 사용해 앞뒤 공백을 절삭합니다.
- **타입 에러 방지**: `int()` 형변환을 시도하고 문자가 들어왔다면 `ValueError`를 발생 및 캐치(`except`)하여 재입력을 요청합니다.
- **빈 입력 체킹**: Enter만 눌러서 Empty String(`''`)이 들어오면 루프를 돌아 다시 입력을 재촉합니다.
- **Out of Range 방지**: 주어진 매개변수 사이(`min` <= `val` <= `max`) 값인지 명시적으로 확인하고 범위를 벗어나면 에러 메시지와 함께 재차 묻습니다.

### 🔌 안전 종료 기능 (Ctrl+C / EOF)
만약 콘솔 게임 중에 사용자가 강제로 인터럽트(Ctrl+C)를 날리거나 파이프가 끊겨 EOF 에러가 발생해도, `main` 루프의 `try - except KeyboardInterrupt, EOFError` 블록에서 에러를 낚아채서 `self.save_data()`를 선행 실행한 뒤 "안전하게 저장하고 종료합니다"라는 메시지와 함께 프로그램을 셧다운합니다.

## 4. 데이터 영속성과 파일 처리 메커니즘 (state.json)

### ❓ JSON 형식을 데이터 스토리지로 선택한 이유
- 파이썬의 표준 `json` 라이브러리로 쉽게 Python Dict / List 객체들과 상호 직렬화/역직렬화할 수 있습니다.
- 사람이 직접 열어서 내용을 읽고 수정하기도 편하며(`가독성`), 구조적인 트리 포맷형식입니다.

### 🧱 중첩 스키마 구조 설계 원인
퀴즈 목록(List), 단순 스칼라 값인 최고점수(Int), 히스토리 목록(List)까지 다양한 타입의 데이터를 파일 하나에서 전부 통합적으로 관리하기 위해, 최상위에 `{ "quizzes": [...], "best_score": ...}`와 같은 하나의 오브젝트로 묶는 중첩(Nested) 계층 구조로 구성했습니다.

### 💾 읽기/쓰기 동작 흐름 (Data Flow)
- **읽기 흐름**: `main.py`에 의해 `QuizGame`이 초기화(`__init__`)되는 맨 처음 딱 한 번만 파일 파싱을 수행합니다.
- **쓰기 흐름**: 게임 환경에 수정(퀴즈 추가, 퀴즈 삭제, 퀴즈 플레이 완주로 점수 기록 갱신)이 일어나는 특정 이벤트가 끝난 직후, 또는 프로그램이 정식 종료/강제 종료될 때마다 `self.save_data()`를 호출하여 파일을 최신으로 덮어씁니다.

### ⚠️ 파일 손상 및 I/O 권한 에러 처리 (try - except 체계)
파일 스토리지 접근 시 예측 불가의 인프라 에러가 터질 확률이 있으므로 방어 코드를 갖추었습니다.
- **파일이 외부적인 요인으로 깨지거나 망가진 경우**: 역직렬화 중 떨어지는 파이썬 `JSONDecodeError`를 잡아서 사용자에게 무너진 데이터임을 알리고, 시스템에 미리 하드코딩해둔 **기본 제공 퀴즈 7개**(7개 충족 완료)로 상태를 복원(Fallback) 시킵니다.
- **I/O 권한이 부족한 경우 (IOError)**: 저장 시도 중 에러가 생기면 예외 메시지만 찍고 앱을 패닉에 빠뜨리지 않게 합니다.

### 📊 지속 성장 시 JSON 방식의 기술적 한계
- JSON 모듈은 파일 데이터를 읽을 때 **통째로 메모리에 로드**하여 메모리 점유율을 끌어올리고, 단 하나의 문제를 수정하더라도 **전체 파일을 덮어써야** 하는 비합리성을 가집니다. 데이터 수가 수천 개로 늘어난다면 디스크 I/O 응답 병목이 생깁니다.
- 특정 ID의 문제만 필터링하거나 O(1)의 속도로 Index 탐색을 지원하지 않습니다. 따라서 퀴즈 볼륨이 거대해진다면 `SQLite` 등의 RDBMS나 `SQLAlchemy` ORM으로의 마이그레이션이 필요합니다.

### ✍️ 커밋 메시지 컨벤션 및 단위
모든 코드를 전부 만들고 한 번에 All-Push를 하는 것이 아니라, 맥락을 역추적할 수 있게 **의미 있는 단위**(모델 생성 -> 메뉴 뼈대 생성 -> 플레이 메인 기능 개발 ... )로 잘게 쪼개어 다음과 같은 Conventional Commits 형태를 지켰습니다.
- `Init`: 최초 뼈대 스켈레톤, 환경설정
- `Feat`: 신규 기능 개발 및 로직 반영
- `Docs`: 문서 수정

### 🔀 브랜치 구성 (Branch Workflow)
`feature/play-quiz`라는 독립된 브랜치를 생성하여 개발한 뒤 `main`으로 Merge 했습니다.
